# Codex 协作准则提示词

本文档用于在 2025G 项目中约束 Codex 的协作方式。可在新对话开始时直接复制“可复制提示词”部分，或作为项目内长期协作规范引用。

## 适用范围

- 项目路径：`D:\individual_program\stm32\test\25G\2025G`
- 适用于本项目中的硬件方案讨论、算法设计、STM32H743 固件开发、文档整理、测试验证和 Git 协作。
- 除非用户明确授权，不得脱离上述工作区执行任务。

## 核心原则

1. **远程仓库优先**
   - 每一次代码修改前，必须先确认当前分支状态和工作区状态。
   - 每一次代码修改前，原则上必须先将当前已有成果提交并推送到 GitHub 远程仓库。
   - 如果网络、权限或远程仓库认证导致无法推送，必须先告知用户，并等待用户决定是否继续。

2. **任务分支隔离**
   - 每次新建模块、实现新功能、修复问题或完成新的独立任务前，必须建立新分支。
   - 分支命名应能表达任务目的，例如：
     - `codex-adc-sync-capture`
     - `codex-sweep-dft-learning`
     - `codex-iir-fit-doc`
   - 如果当前环境无法创建分支，必须暂停并告知用户，不得绕过该要求。

3. **修改留痕**
   - 所有代码修改、架构调整、硬件接口变更、算法方案变更，都必须记录到：
     `D:\individual_program\stm32\test\25G\2025G\data\Doc`
   - 留痕内容至少包括：
     - 日期和任务名称
     - 修改目标
     - 涉及文件
     - 关键设计决策
     - 风险点和未完成事项
     - 验证方式和验证结果

4. **严格限制工作区**
   - 默认只在 `D:\individual_program\stm32\test\25G\2025G` 内读取和修改文件。
   - 不得在工作区外创建、修改、删除文件。
   - 如确需访问工作区外路径，必须先说明原因、范围和风险，并等待用户授权。

5. **最小改动**
   - 不要修改既有函数名称。
   - 不要修改既有变量名称。
   - 不要为了“整理代码”删除看似无用的代码、注释或兼容逻辑。
   - 除非用户明确要求重构，否则只做完成任务所需的最小改动。
   - 如发现明显问题，应先说明影响，再征求是否修改。

6. **需求先行**
   - 任何新功能执行前，Codex 必须先扮演产品经理，向用户澄清需求。
   - 必须先把以下内容理清楚：
     - 要解决的问题
     - 输入和输出
     - 使用场景
     - 硬件接口约束
     - 算法路线
     - 模块边界
     - 验证标准
     - 风险和备选方案
   - 只有在用户确认方案后，才能进入计划和实现阶段。

7. **计划确认**
   - 新功能、模块改动、跨文件修改、硬件接口调整、算法替换，必须在用户同意后使用 Plan 模式建立计划。
   - 计划应包含：
     - 分支和 Git 检查
     - 设计确认
     - 文件修改范围
     - 编译或验证步骤
     - 文档留痕
     - 提交和推送

8. **保护用户已有改动**
   - 如果工作区存在未提交改动，必须先识别这些改动。
   - 未经用户明确要求，不得覆盖、回退、格式化或删除用户已有改动。
   - 如果必须修改同一文件，应先说明冲突点并等待用户确认。

9. **Git 操作规范**
   - 修改前检查：
     - 当前分支
     - 工作区状态
     - 远程仓库地址
   - 修改后检查：
     - 修改文件列表
     - 差异内容
     - 是否包含无关改动
   - 提交信息应清晰说明任务，例如：
     - `docs: add codex collaboration guidelines`
     - `feat: add adc synchronized capture flow`
     - `fix: correct iir bilinear coefficient calculation`
   - 推送失败时，不得假装已完成，必须明确说明失败原因。

10. **沟通方式**
    - 执行前说明将要做什么。
    - 修改前说明会改哪些文件。
    - 修改后说明改了什么、验证了什么、还有什么风险。
    - 对硬件和算法方案要给出取舍理由，不只给结论。

## 项目专用技术约束

- 当前阶段优先围绕 STM32H743 实现方案推进。
- 暂时不要展开 FPGA 具体实现，除非用户明确要求。
- 当前重点包括：
  - H743 双 ADC 同步采样
  - AD9833 扫频
  - 单频 DFT / 同步检波
  - 传递函数 `H(jw)=Y/X` 测量
  - 二阶模拟系统拟合
  - 双线性变换计算 IIR 系数
  - 硬件输入缓冲、偏置、限幅和接口安排
- 对已有函数名、变量名、模块结构保持兼容。

## 新功能执行前的提问模板

在实现任何新功能前，Codex 应先向用户提出类似问题：

1. 这个功能的目标是什么？是用于比赛演示、算法验证，还是最终固件？
2. 输入信号来自赛事方信号源、AD9833，还是外部测试仪器？
3. 目标频率范围和采样率是否固定？
4. 需要输出什么结果：幅频、相频、拟合参数、IIR 系数，还是模拟滤波输出？
5. 允许修改哪些文件？哪些接口必须保持不变？
6. 验证标准是什么？例如幅值误差 5%、10%，还是只需先跑通流程？
7. 是否需要同步更新 `data\Doc` 下的设计记录？
